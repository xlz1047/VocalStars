The 3-File Pattern (every feature folder)

🧮

extractor.py — Pure DSP, no PyTorch

librosa + numpy only. Runs without a GPU. Used for labelling training data and as a fallback when no model is loaded.

🧠

model.py — PyTorch nn.Module head

The learnable head (Linear / Conv layers). Inputs are backbone embeddings. Outputs are logits or probabilities.

🔌

detector.py — Combines both, called by pipeline.py

Tries the model path first; falls back to DSP extractor. This is the only file pipeline.py imports — it hides whether we have a trained model or not.

📡 The 3 Output Signals — What They Mean

What is pitch_hz per frame?

The model outputs a pitch value in Hz for every 10ms frame of audio. Hz = Hertz = vibrations per second — the physical unit of frequency that our ears perceive as pitch.

Middle C on a piano is 261.6 Hz. The A above it is 440 Hz. An octave higher doubles the frequency.

In VocalStars: pitch_hz per frame = "what exact note is the singer hitting right now, 100 times per second"

This is not a single number for the whole clip. It's an array — one Hz value per 10ms window. Over a 3-second clip, you get 300 values, one per frame.

How is it computed?

1. Model outputs pitch_logits (B, T, 360) — raw scores for 360 frequency bins

2. sigmoid() converts to probabilities → a "posteriorgram"

3. Viterbi decoding finds the smoothest path through the posteriorgram over time (avoids random frame-to-frame jumps)

4. The winning bin index is converted to Hz: 31.7 × 2^(bin × 20 / 1200)

The 360 bins: 20 cents each, from B0 (31.7 Hz) to ~8kHz. 20 cents ≈ 1/5 of a semitone — finer than human pitch discrimination (~10 cents).

Pitch stability score

The detector computes: 1 − std(voiced_frames) / mean(voiced_frames)

A score of 1.0 = perfectly stable (no variation). 0.0 = all over the place. This feeds directly into the coaching feedback.

Each bar is one of 360 bins (20 cents wide). The model outputs the probability that the current frame's pitch falls in each bin. Viterbi picks the best path, not just the highest bar.

What is onset_probability per frame?

An onset is the moment a new note or syllable begins. The model outputs a probability (0–1) at every 10ms frame answering: "is this the start of a new note?"

Think of it like this: If you clap once per beat while someone sings, each clap marks an onset. The model is learning to do this automatically.

Onset detection is crucial for rhythm coaching — it lets the system measure whether the singer is hitting notes on the beat, early, or late.

Why is it sparse?

In a typical 5-second singing clip at 10ms per frame = 500 frames total. Maybe 15–25 frames are actually onsets. That's only ~4% of frames.

This is called class imbalance. Without correction, the model learns "always predict 0" which is 96% accurate but completely useless.

Fix: the onset loss applies a 10× weight to positive (onset) frames — the model is penalised 10× harder for missing an onset than for a false alarm.

Training label source

During training, onset labels come from three sources depending on the dataset:

• GTSinger: phoneme ph_start times from JSON sidecars → converted to frame indices

• TextGrid files: phoneme boundaries parsed by the textgrid library

• Fallback: librosa.onset.onset_detect() — DSP heuristic based on spectral flux

RhythmHead Architecture

# input: (B, 384, T) from backbone
x = ReLU(Conv1d(384→128, k=3, pad=1))
x = ReLU(Conv1d(128→64,  k=3, pad=1))
x = sigmoid(Conv1d(64→1, k=1))
# output: (B, 1, T) onset probs per frame

The 1×1 final conv is a per-frame linear classifier on 64-dim features. No pooling = full temporal resolution preserved.

What is breath_bool / breath_prob?

A single probability per audio clip answering: "does this clip start with a breath intake?"

Unlike pitch and onset (which are per-frame), breath is a clip-level detection. The model looks at the whole clip and makes one binary decision.

Why is breath coaching important? Singers who audibly "gulp" before phrases have poor breath support and vocal placement. Identifying it is the first step to fixing it.

How is it detected?

Two-layer approach:

1. DSP heuristic (extractor.py) — used for training labels:

• Silence: RMS < 0.01 → "no breath, just silence"

• Breath: RMS 0.01–0.05 AND ZCR > 0.1 AND centroid < 2000 Hz

• Singing: RMS > 0.05 → "definitely phonating"

2. Neural (model.py) — used at inference:

Backbone features (B, 384, T) → mean(dim=-1) → (B, 384)

→ FC(384→64) → ReLU → FC(64→1) → sigmoid → breath_prob

Why average pooling, not per-frame?

A breath is a sustained ~200ms sound, not a single frame event. Averaging over all T frames collapses the time axis — the model sees the "average character" of the clip, which is sufficient to distinguish "starts with breathy intake" from "starts singing immediately".

Viterbi Decoding — Smoothing the Pitch Track

Problem: The raw model output (posteriorgram) can jump wildly frame-to-frame — the model might say "A4" then "C3" then "A4" across consecutive 10ms frames. That's physically impossible for a real singer.

Solution: Viterbi finds the most likely sequence of pitch states subject to a constraint: pitch can't change more than ±12 bins (±240 cents) per frame.

State space: 360 voiced bins + 1 unvoiced state = 361 total states per frame. The algorithm finds the best path through this state space over time.

Two variants

Offline Viterbi (viterbi_decode): processes all frames at once using backtracing. More accurate — can look ahead. Used for evaluation.

Realtime Viterbi (viterbi_decode_realtime): processes frame-by-frame, left-to-right, no backtracing. Slightly less accurate but works in a streaming context — used in the live phone app.

Onset/offset penalty

Switching between voiced and unvoiced states costs onset_penalty=2.0 in log-space. This makes the model "commit" to voicing — it won't randomly flip unvoiced for a single frame unless the evidence is overwhelming.

Dynamic Programming Intuition

Viterbi is dynamic programming over a trellis. At each frame, instead of asking "what's the best pitch now?", it asks "what's the best pitch now, given the best path to get here?"

Analogy: GPS routing — not just "which road is fastest right now" but "which route gets you there fastest overall, including traffic 3 miles ahead."

YIN Pitch Estimation

YIN is a classical DSP algorithm for pitch detection, implemented in librosa as librosa.yin(). It's used to generate training labels when ground truth pitch isn't available.

How it works

1. Autocorrelation: correlate the signal with a delayed version of itself. Peaks occur at the fundamental period T₀.

2. Difference function: d(τ) = sum of squared differences between signal and its τ-shifted version. Minimum at τ = T₀.

3. Cumulative mean normalisation: removes the bias of d(τ) increasing with τ.

4. Parabolic interpolation: sub-sample precision — finds the minimum between discrete τ samples.

In VocalStars: YIN is the DSP fallback in PitchDetector._analyze_with_librosa() and the label source in all dataset _extract_labels() methods. The trained model learns to do what YIN does, but better.

YIN parameters in the code

# In dataset loaders:
f0 = librosa.yin(
    audio,
    fmin=50,    # 50 Hz = ~G1
    fmax=2000,  # 2 kHz = ~B6
    sr=sr
)
voiced = f0[f0 > 0]          # 0 = unvoiced
pitch_hz = float(np.median(voiced))

Why median, not mean?

A clip might have a few YIN estimation errors (spurious values far from the true pitch). Median is robust to outliers — the middle value of a sorted list is unaffected by extreme values at either end.

Analogy: If 10 students score 85, 88, 90, 91, 92, 93, 94, 95, 97, 3 — the mean is pulled down by the one bad score (81.8), but the median is still 91.

CREPE-style Soft Classification

CREPE (Convolutional REpresentation for Pitch Estimation) is a 2018 paper that showed classifying pitch into bins outperforms direct regression.

Why not just regress Hz directly?

Direct regression: output one number = Hz estimate. Problems:

• Loss surface: mean squared error between 440 Hz and 660 Hz = 48400. Between 440 Hz and 441 Hz = 1. The model "cares" more about large errors even when the perceptual difference is small (one Hz vs one semitone ≈ 25 Hz).

• Uncertainty: no way to express "I think it's A4 or A5 but not sure."

The soft Gaussian target

Instead of a hard one-hot label, we use a Gaussian bell centred on the target bin:

target[b] = exp(−0.5 × ((cents_to_bin − b) / sigma)²)
# sigma = 1 bin = 20 cents

This means adjacent bins get partial credit — the loss is smooth near the correct pitch.

Hz recovery at inference

probs = sigmoid(pitch_logits)  # (T, 360)
hz = (probs × BINS_HZ).sum(−1) /
     (probs.sum(−1) + 1e-8)
# weighted mean over bin frequencies

If probability spreads across two adjacent bins, the Hz estimate naturally falls between them — interpolated for free.

Causal Convolution

A standard 1D convolution with kernel size k=3 and "same" padding looks at frames [t−1, t, t+1] — it needs one frame from the future. This is fine for offline processing but adds latency to streaming.

Causal padding replaces symmetric padding with left-only padding:

# Normal (non-causal):
F.pad(x, (1, 1))  # pad 1 each side
Conv1d(k=3)      # sees t−1, t, t+1

# Causal:
F.pad(x, (2, 0))  # pad 2 on LEFT only
Conv1d(k=3)      # sees t−2, t−1, t
Result: Frame t's output depends only on frames ≤ t. No lookahead. No added latency. The model can produce output the instant a new frame arrives.

Streaming buffers

For frame-by-frame inference, NanoPitchEncoder.init_streaming_state() returns conv1_buf and conv2_buf — the 2-frame sliding windows that the causal conv needs. You just shift these buffers each frame instead of recomputing from scratch.

GRU — Gated Recurrent Unit

A GRU maintains a hidden state h that acts as the model's "working memory". Two gates control how this memory updates each frame:

🔄

Update gate (z)

How much of the old memory to keep vs replace. z=1 → keep old state entirely. z=0 → replace with new input.

🗑️

Reset gate (r)

How much of the old memory to use when computing the new candidate. r=0 → ignore old state (fresh start for this input).

These gates let the GRU learn what to remember and forget — e.g., remember the key/tempo, forget transient noise.

Why 3 stacked GRUs? Each layer operates on the previous layer's output. GRU1 captures short-term patterns (single phoneme), GRU2 captures syllable-level, GRU3 captures phrase-level context.

Orthogonal init

The recurrent weight matrix W_hh is initialised so its singular values are all 1.0.

This prevents the "vanishing gradient" problem: when gradients are backpropagated through many time steps, repeatedly multiplying by W_hh can make them shrink exponentially (vanish) or grow (explode).

An orthogonal matrix has eigenvalues on the unit circle — it preserves gradient magnitudes through recurrence.

# In backbone.py _init_weights:
for name, param in gru.named_parameters():
    if "weight_hh" in name:
        nn.init.orthogonal_(param)

vs LSTM

LSTMs have 3 gates and a separate cell state. GRUs have 2 gates, are ~25% fewer parameters, and typically match LSTM performance on short sequences. Better fit for a mobile-constrained model.

Focal Loss & Weighted BCE

Standard binary cross-entropy treats every sample equally. This is fine when classes are balanced. In singing analysis they're not:

Weighted BCE (onset loss)

Simply multiply the loss by 10 for positive frames: loss × pos_weight. Easy, effective.

weights = ones_like(onset_targets)
weights[onset_targets > 0.5] = 10.0
loss = (weights × BCE(pred, target)).mean()

Focal BCE (VAD loss)

Focal loss additionally down-weights easy examples:

pt = pred if target==1 else (1−pred)
focal_weight = (1 − pt)^gamma
loss = focal_weight × pos_weight × BCE

When the model is very confident (pt→1), (1−pt)^γ ≈ 0 — easy examples barely contribute. Hard examples (pt≈0.5) dominate training. This focuses learning on the boundary cases.

Int8 Quantisation

The trained model uses 32-bit floating point weights (fp32). Each parameter = 4 bytes. 541K params × 4 = ~2.1 MB.

Int8 quantisation maps every weight from a float to an 8-bit integer (0–255) using a per-layer scale factor:

w_int8 = round(w_float32 / scale)
scale = max(|w|) / 127

Reduces: 4 bytes → 1 byte per weight = 4× compression → ~0.5 MB.

Accuracy impact

Quantisation error is small if the weight distribution is well-behaved (no extreme outliers). For well-trained models it typically costs <1% accuracy.

Hardware benefit: Most modern phones have INT8 SIMD instructions that run 4× faster than fp32 on the same operations. So quantisation gives ~4× speed AND ~4× size reduction.

🔄 Full Data Flow — From Singing to Coaching Feedback

1. Microphone input

16 kHz mono audio, captured in 256-sample frames (16ms) with 50% overlap → 8ms latency per output.

2. MelExtractor.compute_tensor()

STFT (400-sample window, 160-sample hop) → 40 mel filter bands → log(mel + 1e-9). Output: (1, 40, T) float32 tensor.

3. NanoPitchEncoder.forward()

Causal conv block (40→96) + 3× GRU (96→96 each) + skip-concat. Output: (B, T, 384) frame embeddings. GRU hidden states h1/h2/h3 are passed to the next call.

4. PitchHead.forward() + logits_to_hz()

Linear(384→360) → pitch_logits. Sigmoid → posteriorgram. Realtime Viterbi → smooth f0_hz array. Linear(384→1) → VAD probability.

5. RhythmHead.forward()

Features permuted to (B, 384, T). Conv1d stack → per-frame onset_probs. Threshold at 0.5 → onset frame indices.

6. BreathHead.forward()

Features averaged over T → (B, 384). FC → breath_prob scalar. Threshold at 0.5 → breath_bool.

7. coaching_engine/scorer.py

Converts raw signals to 0–1 scores: pitch stability from std(f0), rhythm from timing variance of onsets, breath support from breath_prob.

8. coaching_engine/feedback.py

Maps scores → actionable text. score < 0.7 on pitch → "Slow sustained scales". score < 0.7 on rhythm → "Practice with metronome". Returns summary + exercises.

🏗 Design Philosophy — Why Is the Code Structured This Way?

Why extractor / model / detector in every folder?

Each feature (pitch, rhythm, breath) has this same 3-file pattern:

extractor.py — Pure DSP

No PyTorch, no GPU. Runs on any machine. Used to generate training labels from raw audio. Can run as a standalone tool. This is the "ground truth generator."

model.py — nn.Module head

Only the learnable head layers. Takes backbone embeddings as input. Has no awareness of audio loading or DSP. Easily exported to ONNX.

detector.py — Unified interface

Takes raw audio. Tries the model path if a trained model exists; falls back to DSP. pipeline.py only ever imports from detector.py — it doesn't care which path runs.

The payoff: You can develop and test extractor.py without any ML infrastructure. You can swap the model without changing pipeline.py. The DSP fallback means the app works even before training completes.

Why is _model/ separate from the feature folders?

The backbone + full model live in _model/ (note the underscore prefix), separate from pitch/rhythm/breath folders.

📦

Shared backbone

NanoPitchEncoder is used by all three heads. Putting it in pitch_detection/ would be wrong — rhythm and breath also depend on it.

🔒

Underscore convention

The leading underscore in _model/ signals "internal module — not a public API". Feature folders (pitch_detection/, etc.) are the public API that pipeline.py uses.

🏋️

Training vs inference

training/ and _model/ are tightly coupled. Everything else (feature folders, data, export) doesn't need to know training details.

Why abstract SingingDataset?

Seven different datasets (VocalSet, GTSinger, MIR-1K, CSD, DSing, NUS-48E, PopBuTFy) all have different directory structures, annotation formats, and naming conventions. But training code needs them to behave identically.

The abstract base class SingingDataset defines the contract: every subclass must implement _get_filepaths() and _extract_labels(). The base class owns everything else: audio loading, mel computation, __getitem__, __len__.

Adding a new dataset: Subclass SingingDataset, implement the two abstract methods, add it to _build_datasets() in train.py. Nothing else changes. 
# To add a new dataset:
class MyDataset(SingingDataset):
    def _get_filepaths(self) -> list[dict]:
        # return [{"audio_path":..., ...}]
        ...
    def _extract_labels(self, path, meta):
        # return {"pitch_hz":..., ...}
        ...

Why singer-level train/val/test split?

If you split randomly by clip, a singer who appeared in training also appears in validation. The model can "memorise" their voice characteristics — recognise the timbre, not the skill.

Singer-level split ensures validation tests on voices the model has never heard. This is the only honest way to estimate generalisation to new users.

Analogy: Testing a voice coach on a student they've never met, not on a recording of a student they trained. If they can only coach familiar students, they're not a real coach.

Why CosineAnnealingLR + warmup?

🌡️

Warmup (first 500 steps)

At random init, gradients are huge. Starting with a tiny LR prevents the first few batches from causing catastrophic weight changes that take many epochs to recover from.

🌊

Cosine annealing

LR follows a cosine curve from peak to near-zero. The model starts exploring broadly (high LR) then gradually commits to a solution (low LR). Better final loss than linear decay in practice.

🧮 Math & Formulas — How Each Head Transforms Features

Step 1 — Linear projection to 360 bins

pitch_logits = Wpitch · x + bpitch
where   x ∈ ℝ384  (one frame of backbone features)
       Wpitch ∈ ℝ360 × 384  (learnable weight matrix)
       bpitch ∈ ℝ360  (learnable bias)
       output ∈ ℝ360  (raw logit per pitch bin)

Each of the 360 output values is a dot product of the 384-dim feature vector with one row of W — asking "how much does this feature pattern look like it belongs to bin b?"

Parameter count: 384 × 360 = 138,240 weights + 360 biases = 138,600. This is why PitchHead dominates the param budget.

Step 2 — Sigmoid to probabilities

probsb = σ(pitch_logitsb) = 1 / (1 + e−logitsb)
applied element-wise across all 360 bins → probs ∈ [0,1]360

Step 3 — Bin frequencies

Hzb = 31.7 × 2(b × 20) / 1200
31.7 Hz = B0 (lowest bin)  ·  20 cents per bin  ·  1200 cents per octave
b = 0 → 31.7 Hz, b = 60 → 126 Hz, b = 180 → 392 Hz (G4), b = 360 → ~8 kHz

Step 4 — Weighted mean → pitch_hz

pitch_hz = (Σb=0359 probsb × Hzb) / (Σb=0359 probsb + ε)
ε = 1e-8 prevents division by zero  ·  output: one Hz value per frame

The weighted mean naturally interpolates between bins. If probs peak between bin 180 and 181, the output Hz falls between their frequencies — no rounding to the nearest bin needed.

Dimension Flow — PitchHead

Conv1d — The Core Operation

yc_out, t = Σc_in Σk=0K−1 Wc_out, c_in, k · xc_in, t+k−⌊K/2⌋ + bc_out
K = kernel size (3)  ·  c_in = input channels  ·  c_out = output channels
The kernel W slides over the time axis t, summing over all input channels

Think of the kernel as a sliding template: for each output time step, it takes a weighted sum of K consecutive frames across all input channels.

Full RhythmHead forward pass

Input: x ∈ ℝ384 × T
z₁ = ReLU(Conv1d384→128, k=3(x)) → ℝ128 × T
z₂ = ReLU(Conv1d128→64, k=3(z₁)) → ℝ64 × T
z₃ = Conv1d64→1, k=1(z₂) → ℝ1 × T
onset_probs = σ(z₃) → ℝ1 × T, each value ∈ [0,1]

The k=1 final layer is a per-frame linear classifier — each time step independently decides "onset or not?" based on its 64-dim local feature.

Why k=3 not k=1 for the first two? k=3 lets each output frame see ±1 neighboring frames of context — enough to detect the onset envelope shape without needing long-range memory (the backbone's GRUs handle that).

Dimension Flow — RhythmHead

Step 1 — Temporal Average Pooling

pooledf = (1/T) × Σt=1T xf, t
x ∈ ℝ384 × T  (backbone features, channel-first)
pooled ∈ ℝ384  (one value per channel, averaged over all T frames)
T can be any length — pooling makes the head input-length invariant

This collapses the time dimension entirely. A 5-second clip and a 2-second clip both produce the same 384-dim pooled vector shape — the head sees the average character of the clip.

Step 2 — First FC layer with ReLU

h = ReLU(W₁ · pooled + b₁)
W₁ ∈ ℝ64 × 384  ·  b₁ ∈ ℝ64
ReLU(z) = max(0, z)  applied element-wise
h ∈ ℝ64  (compressed representation of breath character)

Step 3 — Output layer + Sigmoid

logit = W₂ · h + b₂
breath_prob = σ(logit) = 1/(1 + e−logit)
W₂ ∈ ℝ1 × 64  ·  b₂ ∈ ℝ1
breath_prob ∈ [0, 1]  ·  threshold at 0.5 → breath_bool
Full formula (composed):
breath_prob = σ( W₂ · ReLU(W₁ · mean_t(x) + b₁) + b₂ )

Sigmoid — logit → probability

z >> 0 → prob ≈ 1.0 (definitely breath). z << 0 → prob ≈ 0.0 (no breath). Decision boundary at z = 0 → prob = 0.5.

Why do we need activation functions?

Without activations, stacking multiple linear layers (y = Wx + b) is mathematically equivalent to just one linear layer. No matter how deep you go, you can always collapse it to a single matrix multiply.

Non-linear activations break this — they let the network learn curved decision boundaries and complex feature combinations.

Proof (without activation): y = W₂(W₁x + b₁) + b₂ = (W₂W₁)x + (W₂b₁ + b₂) = W′x + b′ → still linear.

Sigmoid σ(z) = 1/(1+e−z)

Used in: All three heads for output probabilities.

Range: (0, 1) — maps any real number to a probability.

Gradient: σ′(z) = σ(z)(1 − σ(z)) — peaks at z=0 (0.25), vanishes for large |z|.

Saturation problem: For very large |z| the gradient ≈ 0 — gradient can't flow. This is why sigmoid is only used at the output layer in VocalStars, not in hidden layers.

ReLU max(0, z)

Used in: RhythmHead and BreathHead hidden layers.

Range: [0, ∞) — kills negative activations.

Gradient: 1 for z > 0, 0 for z < 0. No saturation for positive values.

Sparsity: Roughly half the neurons output zero at any given time — this acts as implicit regularisation and makes the network efficient.

Why ReLU in hidden layers, sigmoid at output? ReLU's gradient doesn't saturate for positive values → better gradient flow during training. Sigmoid is reserved for outputs where we specifically need probabilities ∈ [0,1].

Tanh (backbone Conv layers)

Range: (−1, 1) — symmetric around zero. Better than sigmoid for hidden layers since outputs are zero-centered, reducing the zig-zag gradient updates.

Formula: tanh(z) = (ez − e−z) / (ez + e−z)
