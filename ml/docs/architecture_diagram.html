NanoPitchEncoder — Shared Backbone

Architecture

Input
(B, T, 40) log-mel, time-major
Causal Conv Block
pad(2,0) → Conv1d(40→64, k=3) → tanh
pad(2,0) → Conv1d(64→96, k=3) → tanh
conv_out: (B, T, 96)
Causal padding: each frame only looks back 2 frames — no future peeking. Enables real-time streaming.
Recurrent Stack
GRU1(96→96) → g1
GRU2(96→96) → g2
GRU3(96→96) → g3
GRU hidden state = the model's "memory" of what it heard. Passed frame-by-frame during streaming.
Skip Concatenation
cat([conv_out, g1, g2, g3], dim=−1)
features: (B, T, 384)
Why concat all 4? conv_out = local texture (this frame), g1 = short context, g3 = long context. Concat lets each head pick what it needs.

Key Techniques

🔒

Causal Convolution

Standard Conv1d with symmetric padding would let the model "see the future". Instead, F.pad(x,(2,0)) adds zeros only on the left, so frame t uses only frames t−2, t−1, t.

Analogy: Like reading a sentence word-by-word without skipping ahead — you can only use what you've heard so far.
🔄

GRU (Gated Recurrent Unit)

GRUs maintain a hidden state h that summarises everything heard so far. Two gates control this: reset (how much to forget) and update (how much new info to add).

Analogy: Like a musician who remembers the tempo and key from the last several bars, not just the last note.
⊕

Orthogonal Init on Recurrent Weights

GRU recurrent matrices are initialised so their singular values are all 1.0. This keeps gradients from shrinking (vanishing) or exploding during backpropagation through long sequences.

Analogy: Like tuning a phone to exact volume before a concert — prevents signal distortion at the start of training.
🏁

NanoPitch Pre-training

This backbone is architecturally identical to NanoPitch (a small open-source pitch tracker). We can load its pre-trained weights directly — the model already "knows" how pitch relates to mel patterns before training starts.

Analogy: Transfer learning — hiring someone who already knows music theory and just needs to learn to coach.

Task Heads — What Each Head Predicts

🎵

PitchHead

354,920 params · per frame

Output 1 — pitch_logits (B, T, 360): 360 bins spaced 20 cents apart, starting at B0 (31.7 Hz). Each bin = one possible pitch.

Output 2 — vad_logits (B, T, 1): Voice Activity Detection — is the singer actually singing right now?

CREPE-style soft classification: Instead of regressing a single Hz number, the model outputs probability across 360 bins. The final Hz is the sigmoid-weighted average of all bin frequencies.

Analogy: Like a musician tuning a guitar — instead of guessing one frequency, they listen across the whole range and find where energy concentrates. Expressing uncertainty = spreading probability mass across nearby bins.
🥁

RhythmHead

123,137 params · per frame

Output — onset_probs (B, 1, T): At each frame, how likely is it that a new note/syllable started here?

Architecture: Three Conv1d layers (384→128→64→1) with ReLU activations — a lightweight CNN on top of the recurrent features.

10× positive weight: Onsets are rare (maybe 5% of frames). Without this weighting, the model learns "never predict an onset" and achieves 95% accuracy doing nothing useful.

Analogy: Like a metronome detector — most of the time there's just sustained sound, but the model must be extra alert to the moments when a new note begins.
💨

BreathHead

16,513 params · per clip

Output — breath_prob (B, 1): One number per audio clip — did the singer start with a breath?

Temporal average pooling: The backbone produces (B, 384, T) frame features. The breath head collapses the T dimension by averaging — because whether there's a breath is a clip-level property, not frame-level.

Label source: RMS energy of the first 100ms. If it's very quiet (RMS < 0.02), it's labelled as a breath.

Analogy: Like a vocal coach listening to whether their student audibly inhales before starting to sing — a whole-clip binary judgment, not something that changes mid-clip.
📐

Why Share a Backbone?

Multi-task learning

All three heads receive the same (B, T, 384) features from one backbone pass. This is multi-task learning:

1. Efficiency: One forward pass, three predictions simultaneously.

2. Regularisation: The backbone must learn representations useful for pitch AND rhythm AND breath — forces it to learn general vocal features rather than overfitting to one task.

3. Parameter budget: Each head is small because the backbone does the heavy lifting.

Analogy: Like a human ear — one auditory cortex serves all your musical judgments: pitch, rhythm, timbre. You don't have separate ears for each.

Training Pipeline

Step-by-step

1

Load audio from disk

librosa resamples to 16 kHz mono. Each dataset contributes audio + DSP-derived labels (pitch via YIN, onsets via librosa, breath via RMS).

2

Compute log-mel spectrogram

STFT (400 sample window, 160 hop) → 40 mel filter bands → log. Output shape: (1, 40, T).

3

Collate batch + build targets

Zero-pad mels to same T, transpose to (B, T, 40). Build Gaussian pitch bin targets centred on the label Hz. Convert onset frame indices to a dense binary mask (B, T).

4

Forward pass

Backbone → (B,T,384). Three heads run in parallel → pitch_logits, onset_probs, breath_prob.

5

Compute MultiTaskLoss

Weighted sum: 1.0×pitch + 0.5×rhythm + 0.3×breath. Onset loss uses 10× pos-weight for class imbalance.

6

Backward + update

AdamW, grad clip at norm 1.0. Linear LR warmup for first 500 steps, then cosine anneal to 1e-6.

7

Validate + checkpoint

After each epoch, evaluate val pitch loss and pitch accuracy (RPA). Save best_model.pt. Checkpoint every epoch as NNN_loss.pt.

Optimiser Setup

AdamW

Adam with decoupled weight decay (1e-4). Weight decay regularises the model by penalising large weights, but AdamW keeps this separate from the gradient adaptive scaling.

Linear Warmup → Cosine Anneal

LR starts near zero, linearly ramps to peak (5e-4) over 500 steps, then follows a cosine curve down to 1e-6. Warmup prevents large gradient updates at random-init when loss gradients are huge.

Analogy: Warming up a car engine before driving — prevents damage from sudden high load.

Gradient Clipping (max_norm=1.0)

If the gradient vector norm exceeds 1.0, it's scaled down to 1.0. Prevents a single bad batch from causing a catastrophically large weight update.

Singer-level Split

Train/val/test split by singer ID, not randomly by clip. This prevents data leakage — the model can't memorise a specific singer's voice during training and "cheat" on validation.

Analogy: Testing a student on new songs they've never heard, not new recordings of songs they practised.

Intuition Guide — Explain It In Class

🎙️

The Big Picture

This is a real-time vocal coaching system. A singer holds their phone up and sings — the model processes 16ms chunks of audio and continuously outputs: "what note are you singing, are you on the beat, did you just take a breath?"

All of this must happen in under 50ms on a mid-range Android phone, so the whole model is only 541K parameters (comparable to a tiny image classifier).

Contrast: GPT-4 has ~1.8 trillion parameters. This model has 0.5 million — roughly 3 million times smaller.
🌊

Why Mel Spectrograms?

Raw audio is a waveform — 16,000 numbers per second that are hard to learn from directly. A mel spectrogram converts this to a 2D image: time on the x-axis, frequency on the y-axis, brightness = energy.

The mel scale compresses high frequencies because human pitch perception is logarithmic — the difference between 100Hz and 200Hz sounds the same as 1000Hz and 2000Hz to us.

Analogy: Like sheet music — instead of showing every air pressure wiggle, it shows what notes are playing and when. Much easier to learn from.
🧠

Why GRUs, Not Transformers?

Transformers use self-attention: every token attends to every other token — O(T²) memory and compute. At 10ms per frame, 5 seconds of audio = 500 frames = 250,000 attention pairs.

GRUs are O(T) — they process one frame at a time and keep a fixed-size hidden state. Perfect for streaming where you can't wait for the full sequence.

Analogy: Transformer = re-reading the whole book every page. GRU = taking notes as you go and only keeping the summary.
🎯

Why Classify Pitch Instead of Regressing It?

The model outputs 360 probabilities (one per 20-cent bin) rather than a single Hz number. The final Hz is computed as a weighted average.

Advantages: (1) Smooth loss landscape near the correct bin. (2) The model can express uncertainty by spreading probability over adjacent bins. (3) Matches how CREPE (a leading pitch tracker) is trained.

Analogy: When asked "how tall is that building?", you say "probably between 40–50 floors" rather than guessing exactly 47. The uncertainty is information.
⚖️

Class Imbalance in Onset Detection

In a 5-second clip at 10ms frames, there are ~500 frames. Maybe 20 of those are note onsets — just 4%. If the model always predicts "no onset" it gets 96% accuracy but is completely useless.

The fix: give onset frames 10× higher weight in the loss, forcing the model to care about getting them right even though they're rare.

Analogy: A fraud detection system: 99.9% of transactions are legit. Accuracy is the wrong metric — you must weight the rare fraud cases heavily.
🔀

Multi-task Learning Regularisation

Training one network to predict three things simultaneously forces the shared backbone to learn general vocal representations rather than features specific to just pitch or just rhythm.

This acts as a regulariser — the backbone can't overfit to one task because it has to serve all three. In practice, multi-task models often outperform single-task models on each individual task.

Analogy: A music student who practises piano, singing, AND ear training simultaneously develops better general musicianship than someone who only practises one.
📱

Mobile Deployment Path

After training in PyTorch, the model is:

1. Exported to ONNX (Open Neural Network Exchange — a portable format)

2. Converted to TFLite (TensorFlow Lite for mobile)

3. Int8 quantised — weights shrink from 32-bit floats to 8-bit integers (~4× compression, ~4× faster on hardware with int8 acceleration)

Target: <5 MB model, <50ms latency on mid-range Android.

Analogy: Like compressing a high-res photo for a website — you lose a tiny amount of quality but the file is 4× smaller and loads 4× faster.
📊

Dataset Strategy

The model is trained on multiple singing datasets simultaneously:

• VocalSet — professional singers, controlled vocal techniques (belt, vibrato, breathy…)

• GTSinger — multilingual, technique-labelled, with phoneme alignments in TextGrid files

• MIR-1K, CSD, DSing, NUS-48E — when available, for diversity

Split is by singer, not by clip — ensures the model generalises to new voices, not just new songs.

Analogy: Training on many different singers is like teaching a vocal coach who has worked with hundreds of students — they learn to hear across accents, ranges, and styles.
