Five specialist coaches sharing the same pair of ears.

The backbone is the shared ears — it listens to your singing and produces 128 numbers every 10 milliseconds. Five specialists then look at those numbers simultaneously:
  • Pitch coach — which note are you singing?
  • Voice detector — singing or silent?
  • Breath coach — is this a breath moment?
  • Rhythm coach — is this the start of a new note?
  • Technique coach — belt, vibrato, falsetto…? (not working well yet)
The whole model has 260,602 numbers (parameters) to learn from data — compact enough to eventually run on a phone. After the heads give their answers, a Viterbi decoder smooths the pitch track so it doesn't jump erratically between frames.

HCQT vs Log-Mel — Motivation (Not Yet Ablated)
This comparison has not been formally run yet. It is on the roadmap for next week.

Why we chose HCQT: Standard log-mel spectrograms mix harmonics together and are not pitch-equivariant — the same note sung by a tenor vs soprano produces a different log-mel pattern. HCQT stacks 6 copies of the CQT at harmonic frequencies (h×f₀), so the harmonic pattern appears at the same bin offset regardless of the fundamental. This makes pitch patterns translation-invariant in the frequency axis, giving the GRU structured pitch information rather than raw filter bank energies.

What we know from the old ml/ module: The predecessor used a 40-band log-mel spectrogram. ml_new switched to HCQT as the first architectural change. Whether this actually improved accuracy over log-mel on the same data is not yet measured.
We slice your singing into 10-millisecond snapshots — 100 per second.

For each snapshot we compute the HCQT: imagine a piano keyboard equalizer, but instead of one bar per key you stack 6 copies — one for the fundamental note, and one each for its overtones. This lets the AI recognise the same note pattern regardless of whether you're a bass or a soprano.

We also measure three simple "is-this-a-breath" signals: loudness, how spread out the energy is (flatness), and how quickly the wave crosses zero. Breath is quiet, airy, and noisy — singing is louder and more tonal.

Note: We haven't actually tested whether HCQT is better than the simpler approach (log-mel) yet — that comparison is next week's work.

Teaching from 3 singing libraries, across 70 rounds of practice.

We used 3 datasets: opera singers from VocalSet (3,613 recordings), multi-technique singing from GTSinger (6,892 recordings), and pop music from PopBuTFy (28,965 recordings — though this one only trained the pitch detector, not the full model).

We carefully kept different singers in different buckets so the model learns to generalise, not just memorise a known voice.

We penalised missing a breath or note onset much more heavily than a false alarm — because in coaching, silently missing a real problem is worse than mentioning one too many.

The model's report card — tested on recordings it had never heard before.
  • 87.3% pitch accuracy — right note within a quarter-tone, 87 out of 100 frames
  • 95.9% voice detection — knows when you're singing vs silent, 96% of the time

Think of the coaching engine as a GPS for your voice.

It figures out where you are (pitch accuracy, breath support, note clarity), compares to where you want to be, and gives you a route — specific exercises to close the gap.

The score formula weighs pitch (50%) most heavily, then voice quality from PRAAT tools (20%), note-start clarity (15%), and breath support (15%).

One important caveat: the technique label (belt, vibrato, etc.) is not reliable yet — it's only right about 16% of the time. We're treating it as a rough hint, not a diagnosis.

The journey from microphone to coaching in 5 steps:
  1. You press Record. The browser shows your live pitch on screen using a simple algorithm.
  2. You press Stop. The audio is packaged into a file and sent to a Python server.
  3. The server feeds it into the neural network, which analyses it frame by frame.
  4. The server sends back a report card: score, issues, and exercises.
  5. The app displays it. Optionally, Google's Gemini AI adds professional coaching language.

07 / What This Can't Do Yet

❌ Technique Classification
16% accuracy for 20 classes. Random guessing = 5%. The model learns a weak signal but cannot reliably distinguish belt from breathy, or vibrato from straight. Root cause: heavy class imbalance (some techniques have <50 training clips), noisy labels from two datasets with overlapping vocabularies, and the technique head architecture is too shallow to capture long-term stylistic patterns from a 2-second window.
❌ HCQT vs Log-Mel Not Measured
We chose HCQT over the predecessor's log-mel spectrogram for theoretical reasons (harmonic translation invariance). Whether it actually improves accuracy on this dataset has not been formally tested. A log-mel baseline experiment is on the roadmap for next week.
⚠️ Viterbi Needs Tuning
Viterbi RPA (89.1%) is worse than argmax RPA (93.1%). The transition penalty is too aggressive for fast-moving pitch content like melisma and glissando. Transition sigma and voicing penalty need re-calibration before Viterbi is worth the extra compute.
⚠️ Limited Dataset Diversity
Trained on GTSinger (English multi-technique), VocalSet (opera), and PopBuTFy (pop). Not validated on: children's voices, elderly singers, non-Western scales, extreme falsetto, or very noisy/reverberant environments. Generalisation to these conditions is unknown.
⚠️ Mobile Export Not Done
The PyTorch model has not been exported to ONNX or TFLite yet. Latency on mobile hardware is unmeasured. The 260K parameter count suggests it should be feasible, but INT8 quantisation and edge benchmarking are still pending.
⚠️ Streaming Not End-to-End Tested
The backbone uses causal convolutions and a causal GRU, so streaming is architecturally supported. However, streaming inference has not been tested end-to-end against the file-based path. Buffering, state management, and latency under real network conditions are unverified.
What the model honestly cannot do yet:
  • Recognise singing style — it tries to detect belt, vibrato, falsetto etc. but is right only 16% of the time. It's barely better than guessing randomly.
  • Proven HCQT advantage — we believe HCQT features are better, but haven't actually run the experiment yet to prove it.
  • Smooth pitch tracking — the Viterbi smoother we built actually makes accuracy worse right now. We're using the simpler approach instead while we tune it.
  • Run on a phone — it hasn't been converted to mobile format yet. The size looks promising, but it's untested.
  • Generalise to all voices — not tested on children, elderly singers, or non-Western music styles.

08 / Next Week

🔬 HCQT vs Log-Mel Ablation
Train a parallel model with identical architecture but log-mel features (n_mels=40, same hop) as input instead of HCQT. Compare RPA, RCA, median_cents, and training convergence speed on the same GTSinger + VocalSet split.

Expected outcome: quantify whether the HCQT's theoretical harmonic-invariance advantage translates to measurable RPA gain on this specific dataset.
🔧 Fix Technique Classification
Current 16% accuracy is not usable. Planned approaches:
  • Audit class distribution — identify severely underrepresented techniques and balance or merge
  • Use a longer input window (from 2 s to 5–8 s) — technique is a long-term feature
  • Decouple from unified model: train a separate AcousticTechniqueClassifier on MFCC + spectral features rather than GRU embeddings
  • Consider reducing to fewer, better-defined classes (e.g. 6–8 broad categories vs 20 fine-grained ones)
Two main tasks for next week:

  1. Run the HCQT vs Log-Mel experiment. Train the same model but with a simpler audio representation, and compare the scores. This will either confirm that HCQT was worth the extra complexity, or reveal we could have done just as well with something simpler.
  2. Fix the technique classifier. 16% for 20 styles is not good enough. The plan is to use longer audio clips (the model currently only sees 2 seconds — not enough to judge singing style), and possibly reduce the number of categories to make the task more learnable.
